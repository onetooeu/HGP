Full integration (Portal Autopilot + ONETOO AI Search Autopilot)

This site is static, but it behaves like a living system: contributions arrive via GitHub Issues, are validated, published to the portal feed, and then prepared for ONETOO indexing (curated accepted-set).

Pipeline (no backend required)

Issue → validate JSON → publish portal feed → publish TFWS trust surface → notify ONETOO maintainers → indexed (accepted-set) → searchable

Goal: get your publication into ONETOO search in a way that is TFWS-verifiable, repeatable, and signature-auditable.

⚠️ ONETOO Standard 2026: public backend is read‑only

search.onetoo.eu currently exposes GET‑only endpoints (e.g. /search/v1) and does not expose a public POST /contrib/…/submit endpoint. Contributions are curated on the ONETOO side (accepted-set), while HGPEdu publishes transparent local audit artifacts (portal feed + integrity).

Test: https://search.onetoo.eu/search/v1?q=hgpedu&limit=10

Roadmap

  • Now: Issue-driven publishing to /portal/index.json (no human step).
  • Next: Automatic ONETOO status sync (no submit endpoint) and local submission ledger /portal/submissions.json.
  • Next: Display ONETOO index status in the portal table (accepted / missing / error).
  • Optional: Signed payloads (Minisign) using GitHub Actions secrets for fully auditable provenance.

Where the data lives

  • Portal feed: /portal/index.json
  • Submission store: /portal/submissions.json

How users contribute

  1. Open a GitHub Issue titled [PORTAL] ...
  2. Paste one JSON object (template is provided)
  3. Autopilot does the rest

Important

ONETOO submission requires an API token. Store it as a GitHub Actions secret (e.g. ONETOO_TOKEN). This keeps the pipeline autonomous while protecting credentials.
